# Лабораторная работа: Паттерн "Компоновщик" (Composite)
## Тема: Система управления квестами в RPG-игре ("QuestManager")

### 1. Легенда и постановка задачи
Вы работаете в студии разработки RPG-игр. Вам поручено создать подсистему управления заданиями (квестами) для игрока. 

**Проблема:** В игре существуют как простые задания (например, "Убить 5 слизней"), так и сложные сюжетные цепочки (например, "Глава 1: Начало пути", которая считается выполненной только тогда, когда выполнены все входящие в неё под-квесты). 

**Задача:** Реализовать систему так, чтобы клиентский код (например, интерфейс игрока или система выдачи наград) работал с одиночными квестами и целыми цепочками квестов **единообразно**, не используя проверки типов (`if is Composite`) и сложные условные конструкции.

### 2. Функциональные требования к приложению
Консольное приложение должно позволять:
1.  **Создавать структуру квестов:** Формировать дерево заданий (Квест -> Подквест -> Под-подквест).
2.  **Отображать иерархию:** Выводить список квестов в консоль с отступами, соответствующими уровню вложенности (визуализация дерева).
3.  **Агрегировать данные:**
    *   Считать **общее вознаграждение** (Золото + Опыт) для выбранного узла. Если выбрана цепочка, сумма должна складываться из всех вложенных квестов.
    *   Считать **общее время выполнения** (в игровых часах).
4.  **Управлять статусом:**
    *   Менять статус выполнения (Выполнено / Не выполнено).
    *   Проверять статус выполнения. Для составного квеста статус "Выполнено" возвращается только если **все** дочерние элементы выполнены.
5.  **Демонстрация пользы:** В методе `Main` создать сложную структуру (минимум 3 уровня вложенности) и показать, что методы вызова награды или проверки статуса работают одинаково для листа и для контейнера.

### 3. Схема классов

Вам необходимо спроектировать и реализовать следующую структуру.

#### 1. Компонент (Interface)
**Имя:** `IQuest`
**Описание:** Общий интерфейс для всех элементов системы квестов.
**Методы:**
*   `string Title { get; }` — Название квеста.
*   `int GoldReward { get; }` — Награда золотом.
*   `int ExpReward { get; }` — Награда опытом.
*   `int EstimatedHours { get; }` — Примерное время выполнения.
*   `bool IsCompleted { get; }` — Статус выполнения.
*   `void Complete()` — Пометить квест как выполненный.
*   `void Reset()` — Сбросить статус выполнения.
*   `void Display(int indentLevel)` — Вывод информации в консоль (с учетом отступа).
*   `void Add(IQuest quest)` — Добавление подквеста (по умолчанию бросает исключение или ничего не делает).
*   `void Remove(IQuest quest)` — Удаление подквеста (по умолчанию бросает исключение или ничего не делает).

#### 2. Лист (Leaf)
**Имя:** `SimpleQuest`
**Реализация:** `IQuest`
**Описание:** Представляет собой элементарное задание, которое не имеет вложенной структуры.
**Поля:**
*   `_title`, `_gold`, `_exp`, `_hours`, `_isCompleted` (приватные).
**Логика:**
*   Конструктор принимает параметры квеста.
*   Методы `Add`/`Remove` должны явно сообщать об ошибке (например, `NotSupportedException`), так как у простого квеста не может быть детей.
*   `Display`: Выводит строку вида `[ ] Убить 10 крыс (Награда: 100 золотых)`. Если выполнено, вместо `[ ]` выводить `[x]`.
*   `Complete`/`Reset`: Меняют локальное поле `_isCompleted`.

#### 3. Компоновщик (Composite)
**Имя:** `QuestChain` (или `QuestGroup`)
**Реализация:** `IQuest`
**Описание:** Представляет собой контейнер, хранящий коллекцию других объектов `IQuest` (как простых, так и других цепочек).
**Поля:**
*   `_title` (Название главы/цепочки).
*   `_children` (Список `List<IQuest>`).
*   `_isCompleted` (Вычисляемое свойство или хранимое поле, обновляемое при изменении детей).
**Логика:**
*   Методы `Add`/`Remove`: Реально добавляют/удаляют объекты из списка `_children`.
*   `GoldReward` / `ExpReward` / `EstimatedHours`: Возвращают **сумму** соответствующих значений всех дочерних элементов. (Сами по себе цепочки могут не иметь награды, либо иметь бонус за завершение главы — на ваше усмотрение, но агрегация обязательна).
*   `IsCompleted`: Возвращает `true` только если список детей не пуст и **все** дети вернули `true` для своего `IsCompleted`.
*   `Complete`: Рекурсивно вызывает `Complete()` у всех дочерних элементов.
*   `Reset`: Рекурсивно вызывает `Reset()` у всех дочерних элементов.
*   `Display`: Выводит заголовок цепочки, затем рекурсивно вызывает `Display(indentLevel + 1)` для каждого ребенка.

#### 4. Клиент (Client)
**Имя:** `Program` (класс `Main`)
**Описание:** Точка входа в приложение.
**Задачи:**
1.  Создать несколько объектов `SimpleQuest`.
2.  Создать объекты `QuestChain`.
3.  Собрать дерево квестов (например, "Сюжетная линия" -> "Глава 1" -> ["Квест А", "Квест Б"] ).
4.  Продемонстрировать полиморфизм: создать список `List<IQuest>`, положить туда и одиночные квесты, и цепочки. Пройтись по списку в цикле и вызвать `Display(0)` и вывести общую награду.
5.  Показать изменение статуса: выполнить один подквест и показать, как изменился статус родительской цепочки.

### 5. Информация для размышления
1.  **Безопасность:** Является ли хорошим решением наличие методов `Add/Remove` в интерфейсе `IQuest`, если у `SimpleQuest` они не работают? (Паттерн "Безопасный Компоновщик" vs "Небезопасный"). 
2.  **Кэширование:** Стоит ли пересчитывать сумму наград каждый раз при обращении к свойству `GoldReward` в `QuestChain`, или лучше обновлять её при добавлении/удалении детей?
3.  **Расширяемость:** Как легко будет добавить новый тип квеста, например, `DailyQuest` (ежедневный), который сбрасывается сам через 24 часа? Потребуется ли менять код `QuestChain` или `Program`?
