# Лабораторная работа: Паттерн "Цепочка Обязанностей" (Chain of Responsibility)

**Дисциплина:** Архитектура ПО на .NET  
**Язык:** C#  
**Тип приложения:** Console Application  

---

## 1. Постановка задачи

Вы разрабатываете бэкенд-модуль для системы обработки заказов интернет-магазина. Бизнес-требования часто меняются: сегодня нужно проверять наличие товара на складе, завтра — добавить проверку на мошенничество, послезавтра — изменить логику расчета налогов.

Использование последовательности `if-else` или жестко закодированных вызовов методов приведет к нарушению принципа Open/Closed и усложнению тестирования. Ваша задача — реализовать гибкий конвейер обработки заказа, используя паттерн **Chain of Responsibility**.

**Суть задачи:**  
Заказ проходит через ряд обработчиков (Handlers). Каждый обработчик выполняет свою конкретную задачу (валидация, скидка, налог, проверка мошенничества). Обработчик может:
1.  Обработать запрос и передать его дальше.
2.  Обработать запрос и **остановить** цепочку (например, заказ отклонен).
3.  Передать запрос дальше без обработки (если задача не относится к этому handler).

Студент должен продемонстрировать возможность динамического изменения порядка и состава цепочки без переписывания клиентского кода.

## 2. Схема классов

Необходимо спроектировать следующую структуру (код писать не нужно, только реализация согласно описанию):

### 2.1. Контекст (Context)
**Класс:** `OrderContext` (рекомендуется использовать `record` или `class`)  
**Назначение:** Хранит данные заказа, которые передаются по цепочке и могут модифицироваться обработчиками.  
**Поля:**
*   `List<OrderItem> Items` — список товаров.
*   `decimal TotalAmount` — итоговая сумма (может меняться после применения скидок/налогов).
*   `bool IsApproved` — флаг успешного прохождения всех проверок.
*   `string RejectionReason` — причина отклонения, если цепочка прервана.
*   `int UserId` — ID покупателя (для проверки мошенничества).
*   `string? CouponCode` — код промокода.

### 2.2. Абстрактный обработчик (Handler)
**Класс:** `Handler` (abstract class)  
**Назначение:** Базовый класс для всех обработчиков. Реализует механизм передачи запроса следующему звену.  
**Поля:**
*   `_nextHandler` (protected) — ссылка на следующий обработчик в цепочке.  
**Методы:**
*   `SetNext(Handler handler)` — устанавливает следующее звено. Возвращает переданный handler для удобной fluent-настройки цепи.
*   `Handle(OrderContext context)` (abstract) — основной метод обработки. Вызывается клиентом.
*   `Process(OrderContext context)` (protected virtual) — шаблонный метод, содержащий общую логику передачи запроса дальше (если `_nextHandler` не null).

### 2.3. Конкретные обработчики (Concrete Handlers)
Реализовать минимум 4 различных обработчика, наследующихся от `Handler`:

1.  **`InventoryCheckHandler`**  
    *   *Задача:* Проверяет, есть ли товары из `Items` в наличии (эмуляция через случайность или хардкод).  
    *   *Логика:* Если товара нет -> ставит `IsApproved = false`, заполняет `RejectionReason`, **обрывает цепочку**. Если есть -> передает дальше.

2.  **`FraudDetectionHandler`**  
    *   *Задача:* Проверяет `UserId` на наличие в "черном списке".  
    *   *Логика:* Если пользователь подозрительный -> отклоняет заказ. Если нет -> передает дальше.

3.  **`DiscountCalculationHandler`**  
    *   *Задача:* Применяет скидку, если в контексте есть валидный `CouponCode`.  
    *   *Логика:* Изменяет `TotalAmount` в `OrderContext`. Не отклоняет заказ, всегда передает дальше (если купон невалиден — просто игнорирует).

4.  **`TaxCalculationHandler`**  
    *   *Задача:* Начисляет налог на текущую сумму заказа.  
    *   *Логика:* Изменяет `TotalAmount` в `OrderContext`. Передает дальше.

*(Опционально)*: **`LoggingHandler`** — просто пишет в консоль текущий этап и состояние контекста, всегда передает дальше.

### 2.4. Клиент (Client)
**Класс:** `Program`  
**Назначение:** Сборка цепочки и запуск обработки.  
**Обязанности:**
*   Создание экземпляров обработчиков.
*   Сборка цепочки в нужном порядке (например: Валидация -> Мошенничество -> Скидка -> Налог).
*   Создание `OrderContext` с тестовыми данными.
*   Запуск обработки через первый элемент цепи.
*   **Важно:** Реализовать меню в консоли, позволяющее пользователю выбрать, какие обработчики включить в цепочку перед запуском (демонстрация гибкости).

## 3. Требования к реализации

1.  **Принцип единственной ответственности (SRP):** Каждый обработчик должен делать только одну вещь (проверять склад, считать налог и т.д.).
2.  **Принцип открытости/закрытости (OCP):** Добавление нового этапа (например, `EmailNotificationHandler`) не должно требовать изменения кода существующих обработчиков или клиента (только сборки цепи).
3.  **Отсутствие жестких связей:** Обработчики не должны знать о конкретных классах других обработчиков, только об абстракции `Handler`.
4.  **Консольный вывод:** Программа должна наглядно выводить лог прохождения заказа:
    *   *"Заказ поступил в обработку..."*
    *   *"[Inventory] Проверка склада... Успешно."*
    *   *"[Fraud] Проверка безопасности... Успешно."*
    *   *"[Discount] Применена скидка 10%. Новая сумма: 900."*
    *   *"Заказ одобрен. Итоговая сумма: 990."*
    *   ИЛИ: *"[Inventory] Товар отсутствует. Заказ отклонен."* (цепочка прерывается).
