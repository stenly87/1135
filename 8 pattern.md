# Лабораторная работа: Паттерн "Строитель" (Builder)
## Тема: Система конфигурации межпланетных исследовательских роверов

### 1. Легенда и контекст
Вы — ведущий инженер в космическом агентстве. Ваша задача — разработать программный модуль для автоматической сборки конфигурации исследовательских роверов перед запуском.

Роботы-роверы — это сложные устройства. Они состоят из шасси, источника энергии, набора научных инструментов (манипуляторы, буры, камеры, спектрометры) и системы связи.
**Проблема:** У каждого ровера есть строгие физические ограничения (лимит массы и лимит энергопотребления). Невозможно просто "накидать" модулей на робота — он либо не взлетит, либо сгорит батарея. Кроме того, объект конфигурации ровера после сборки должен быть **неизменяемым (immutable)**, чтобы исключить случайное изменение параметров во время миссии.

Использование обычного конструктора с 15+ параметрами невозможно (теlescoping constructor anti-pattern), а использование сеттеров опасно (объект может оказаться в невалидном состоянии в процессе сборки).

**Ваша задача:** Реализовать систему сборки роверов, используя паттерн **Builder**, которая гарантирует, что собранный ровер всегда валиден.

---

### 2. Требования к функционалу

1.  **Валидация в процессе сборки:** Строитель должен отслеживать текущую массу и энергопотребление. Если добавление модуля превышает лимиты миссии, сборка должна прерываться или модуль не должен добавляться (с выводом предупреждения).
2.  **Неизменяемость продукта:** Класс `Rover` не должен иметь публичных сеттеров. Все свойства устанавливаются только через Строителя в момент вызова метода `Build()`.
3.  **Различные сценарии сборки (Директор):** Должны быть пресеты для разных типов миссий (например, "Легкий разведчик", "Тяжелый буровой", "Орбитальный челнок").
4.  **Консольный интерфейс:** Пользователь выбирает тип миссии в консоли, система собирает ровера и выводит подробный отчет о конфигурации и проверке лимитов.

---

### 3. Схема классов (Описание)

Необходимо реализовать следующую структуру. **Код писать не нужно, но архитектура должна соответствовать этому описанию:**

#### 3.1. Продукт (Product)
*   **Класс:** `Rover`
*   **Назначение:** Представляет собой готовый конфигурированный ровер.
*   **Свойства (только `get`):**
    *   `ModelName` (string)
    *   `ChassisType` (enum: Wheeled, Tracked, Legs)
    *   `PowerSource` (string: Solar, Nuclear, Battery)
    *   `Modules` (ReadOnlyCollection<string>) — список установленных инструментов.
    *   `TotalWeight` (double) — кг.
    *   `TotalPowerConsumption` (double) — Вт.
*   **Особенность:** Конструктор должен быть `private` или `internal`. Доступ к созданию только через `RoverBuilder`.

#### 3.2. Строитель (Builder)
*   **Интерфейс:** `IRoverBuilder`
*   **Методы:**
    *   `SetChassis(ChassisType type)`
    *   `SetPowerSource(string source)`
    *   `AddModule(string moduleName, double weight, double power)`
    *   `Reset()` — сброс текущего состояния для новой сборки.
    *   `Build()` — возвращает объект `Rover`. Выбрасывает исключение, если ровер не валиден (например, не задано шасси).
*   **Класс:** `RoverBuilder` (реализация `IRoverBuilder`)
    *   Хранит временное состояние (текущий вес, текущая энергия, список модулей).
    *   **Важно:** Внутри метода `AddModule` должна быть логика проверки. У конструктора есть лимиты (например, `MaxWeight = 1000`, `MaxPower = 5000`). Если добавление модуля нарушает лимит, метод возвращает `false` или кидает предупреждение, но не ломает сборку остальных частей.

#### 3.3. Директор (Director)
*   **Класс:** `MissionPlanner`
*   **Назначение:** Знает сложные последовательности шагов для сборки конкретных типов роверов.
*   **Методы:**
    *   `ConstructScoutRover(IRoverBuilder builder)` — собирает легкий ровер (мало веса, камеры, связь).
    *   `ConstructMiningRover(IRoverBuilder builder)` — собирает тяжелый ровер (бур, ядерная батарея, гусеницы).
    *   `ConstructCustomRover(IRoverBuilder builder, Action<IRoverBuilder> configAction)` — позволяет пользователю самому задать шаги через делегат.

#### 3.4. Клиент (Main Program)
*   **Класс:** `Program`
*   **Назначение:** Точка входа.
*   **Логика:**
    1.  Выводит меню выбора миссии.
    2.  Создает `RoverBuilder`.
    3.  Передает строителя в `MissionPlanner`.
    4.  Получает готовый `Rover`.
    5.  Выводит итоговый чек-лист (JSON-подобный формат или красивую таблицу в консоли).

---

### 4. Сценарий использования (User Flow)

1.  Запуск приложения.
2.  Меню:
    *   `1` — Собрать стандартный ровер "Марс-Разведчик" (через Директор).
    *   `2` — Собрать ровер "Астероид-Бур" (через Директор).
    *   `3` — Ручная конфигурация (пользователь вводит названия модулей, система считает вес/энергию в реальном времени).
3.  После выбора:
    *   Консоль показывает процесс: *"Установка шасси... ОК"*, *"Добавление бура... Превышен лимит веса! Модуль пропущен."*.
4.  Финал:
    *   Вывод характеристик готового объекта `Rover`.
    *   Попытка изменить свойство `TotalWeight` у готового ровера должна быть невозможна (проверяется через отсутствие сеттера в коде).

### 5. Бонусные задания (по желанию)

1.  **Логирование сборки:** Добавьте интерфейс `IBuildLogger`. Реализуйте `ConsoleLogger` и `FileLogger`. Строитель должен принимать логгер в конструктор и записывать каждый шаг сборки (успешный или нет) в соответствующее место.
2.  **Клонирование конфигурации:** Добавьте метод `Clone()` в `RoverBuilder`, чтобы можно было взять за основу конфигурацию "Разведчик", добавить к ней один модуль и собрать новую версию, не начиная с нуля.
3.  **Fluent Interface:** Реализуйте методы строителя так, чтобы их можно было вызывать цепочкой:
    ```csharp
    var rover = builder.SetChassis(...)
                       .SetPower(...)
                       .AddModule(...)
                       .Build();
    ```

### 7. Подсказки для студентов
*   Не бойтесь делать класс `Rover` "глупым". Вся логика проверки должна быть в `RoverBuilder`.
*   Подумайте, что должно произойти, если вызвать `Build()` до того, как установлено шасси? (Подсказка: исключение `InvalidOperationException`).
*   Для хранения списка модулей внутри билдера используйте `List<string>`, но в продукте возвращайте копию или `ReadOnlyCollection`, чтобы внешний код не мог изменить список после сборки.
