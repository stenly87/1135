# Лабораторная работа: Паттерн "Стратегия" (Strategy)
## Тема: Система динамического расчета стоимости логистики "LogiCore"

### 1. Легенда
Вы работаете ведущим разработчиком в стартапе **LogiCore**, который предоставляет API для расчета стоимости доставки грузов. Бизнес растет быстро, и каждый месяц маркетологи вводят новые тарифные планы, сезонные надбавки или специальные условия для определенных типов грузов (хрупкие, опасные, скоропортящиеся).

Текущая реализация расчета стоимости представляет собой огромный метод `CalculatePrice` с десятками вложенных `if-else` и `switch`. Любое изменение тарифа требует правки этого метода, что постоянно ломает существующий функционал и усложняет тестирование.

Ваша задача — рефакторить систему, применив паттерн **Strategy**, чтобы сделать добавление новых тарифов безопасным и изолированным процессом.

### 2. Цель работы
*   Понять принцип инкапсуляции вариативного поведения.
*   Реализовать принцип Open/Closed Principle (OCP): система должна быть открыта для расширения (новые тарифы), но закрыта для модификации (существующий код расчета не меняется).
*   Научиться внедрять зависимости через интерфейс стратегии.

### 3. Требования к приложению
1.  **Тип приложения:** Console Application (C#).
2.  **Ввод данных:** Пользователь вводит параметры заказа (вес, объем, расстояние, тип груза) и выбирает тип тарифа из меню.
3.  **Гибкость:** Возможность сменить алгоритм расчета "на лету" для одного и того же заказа (например, сравнить стоимость обычной доставки и экспресс-доставки без пересоздания заказа).
4.  **Запрет:** Использование полиморфизма через наследование от базового класса `Delivery` для разных типов доставки **запрещено**. Поведение должно выноситься в отдельные классы стратегий.
5.  **Запрет:** Использование сложных условий `if (type == "Air" && weight > 10)` внутри контекста.

### 4. Схема классов (Class Diagram Description)

Ниже описана структура, которую необходимо реализовать.

#### 4.1. Интерфейс стратегии (`IFreightStrategy`)
Опредеряет контракт для всех алгоритмов расчета.
*   **Метод:** `decimal CalculateCost(ShipmentOrder order)`
*   **Назначение:** Принимает объект заказа и возвращает итоговую стоимость. Не должен иметь побочных эффектов (не менять данные внутри `order`).

#### 4.2. Контекст (`FreightCalculator`)
Класс, который использует стратегию. Он не знает деталей реализации расчета.
*   **Поле:** `_strategy` (тип `IFreightStrategy`).
*   **Метод:** `SetStrategy(IFreightStrategy strategy)` — позволяет подменить алгоритм во время выполнения.
*   **Метод:** `GetTotalCost(ShipmentOrder order)` — делегирует вызов текущей стратегии.
*   **Метод:** `CompareStrategies(ShipmentOrder order, List<IFreightStrategy> strategies)` — *опционально*, для вывода сравнения цен разных тарифов для одного заказа.

#### 4.3. Класс данных (`ShipmentOrder`)
DTO (Data Transfer Object), содержащий информацию о грузе.
*   **Свойства:**
    *   `double Weight` (кг)
    *   `double Volume` (м³)
    *   `int Distance` (км)
    *   `CargoType Type` (Enum: Standard, Fragile, Hazardous, Perishable)
    *   `bool IsUrgent` (срочность)

#### 4.4. Конкретные стратегии (Concrete Strategies)
Реализуют интерфейс `IFreightStrategy`. Каждая содержит свою уникальную бизнес-логику.

1.  **`StandardGroundStrategy`**
    *   *Логика:* Базовая ставка за км + коэффициент за вес. Если вес > 100 кг, применяется скидка 5%.
2.  **`AirFreightStrategy`**
    *   *Логика:* Высокая базовая ставка. Расчет идет по "объемному весу" (Volume * 167), если он больше фактического веса. Срочность (`IsUrgent`) не влияет на цену, так как это уже авиа.
3.  **`HazardousCargoStrategy`**
    *   *Логика:* Использует логику `StandardGroundStrategy` как базу, но добавляет обязательную страховку (фиксированная сумма) и коэффициент опасности (x1.5). Требует проверки типа груза в заказе.
4.  **`ColdChainStrategy`** (Для скоропортящихся грузов)
    *   *Логика:* Стоимость зависит от расстояния и необходимости поддержания температуры. Добавляется плата за каждый км пути за охлаждение. Если расстояние > 1000 км, добавляется плата за смену хладагента.

#### 4.5. Фабрика стратегий (`StrategyFactory`)
*   **Метод:** `GetStrategy(string tariffCode)`
*   **Назначение:** Возвращает нужный экземпляр стратегии по строковому коду (например, "AIR", "STD", "HAZ"). Избавляет клиентский код (Main) от создания объектов стратегий через `new`.

### 5. Задание на реализацию

1.  **Базовая реализация:**
    *   Создайте интерфейс и все 4 класса стратегий с "заглушками" формул (например, просто `weight * distance * coefficient`).
    *   Реализуйте `FreightCalculator` и `StrategyFactory`.
    *   В методе `Main` создайте тестовый заказ, позвольте пользователю выбрать тариф и выведите стоимость.

2.  **Демонстрация пользы (The "Aha!" moment):**
    *   Реализуйте сценарий, где пользователь вводит данные заказа **один раз**, а программа выводит таблицу со стоимостью доставки этим заказом для **всех доступных стратегий** сразу. Это покажет, что Контекст может работать с разными стратегиями без изменения кода заказа.

3.  **Расширение (Проверка OCP):**
    *   Добавьте 5-ю стратегию `WeekendDeliveryStrategy` (доставка в выходные стоит на 20% дороже).
    *   **Важно:** Вы не должны менять код в классах `FreightCalculator`, `ShipmentOrder` или других существующих стратегиях. Только создайте новый класс и зарегистрируйте его в Фабрике.

### 6. Подсказки для студентов
*   **Не путайте Стратегию с Наследованием.** Не создавайте класс `AirDelivery : Delivery`. Создавайте `AirStrategy : IFreightStrategy`.
*   **Состояние.** Стратегии не должны хранить состояние конкретного заказа. Все данные для расчета передаются через аргументы метода `CalculateCost`.
*   **Зависимости.** Контекст должен зависеть от абстракции (интерфейса), а не от конкретных классов стратегий.
